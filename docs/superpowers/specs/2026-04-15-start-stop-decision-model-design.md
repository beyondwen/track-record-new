# 开始 / 结束记录决策模型设计

## 文档定位

- 文档类型：第一阶段方案设计
- 适用范围：后台自动开始记录 / 自动结束记录决策链路
- 相关纪要：[2026-04-15 对话纪要：开始/结束记录模型方案讨论](/mnt/d/data/AndroidWork/track-record-new/docs/conversations/2026-04-15-start-stop-model-discussion-record.md)

## 1. 摘要

本设计将「是否开始记录」与「是否结束记录」定义为两个独立的边界决策问题，不再以通用大语言模型（LLM）作为主决策器，而是采用「双二分类轻量模型 + 决策平滑器 + 人工纠错闭环」的第一阶段方案。

该方案的核心判断如下：

- 决策目标不是识别所有活动类型，而是稳定地找出「应开始记录」和「应结束记录」两个边界。
- 第一阶段优先减少误结束和误开始，允许开始或结束存在一定延迟。
- 第一阶段模型以 `Logistic Regression` 为基线，必要时再升级到 `XGBoost`。
- 模型只负责输出分数，不直接触发状态切换；最终决策必须经过平滑器和保护规则。
- 系统通过「自动决策后人工纠错」积累训练数据，形成可持续迭代的个人化闭环。

## 2. 背景与问题定义

当前项目已经具备后台自动轨迹记录能力，核心逻辑集中在 `BackgroundTrackingService` 及其周边的运动置信度、噪声过滤、历史落库流程中。现有实现以规则和状态机为主，能够完成基础的自动开始、自动暂停和自动结束，但在边界稳定性和后续可训练性上仍有明显限制。

当前问题不是「系统能不能自动判断」，而是：

- 如何系统性降低误开始和误结束。
- 如何让决策逻辑从一次性规则堆叠，演进为可观测、可回放、可训练、可持续优化的流水线。
- 如何在 Android 端长期后台运行的约束下，选择足够轻量且可维护的实现路径。

因此，本设计的重点不是继续加大规则复杂度，也不是把通用模型塞进后台，而是重构一条适合长期演进的决策链路。

## 3. 设计目标

### 3.1 目标

- 将「开始记录」和「结束记录」拆分为两个独立的最终决策任务。
- 以「少误判，宁可慢一点」作为第一优先级。
- 让决策链路具备可观测、可回放、可训练和可替换能力。
- 支持通过低成本人工纠错逐步积累个人训练数据。
- 为后续模型升级、特征扩展和参数迭代保留稳定接口。

### 3.2 非目标

- 第一阶段不引入通用 LLM 作为主决策器。
- 第一阶段不引入 GPU 训练或 GPU 推理依赖。
- 第一阶段不追求最短触发延迟。
- 第一阶段不要求完全离线完成训练闭环。
- 第一阶段不以兼容当前 `BackgroundTrackingService` 的内部结构为前提。

## 4. 用户偏好与约束

- 关注目标：只关注「开始记录」和「结束记录」两个最终决策。
- 优先级：先减少误结束，再减少误开始，最后再优化延迟。
- 延迟容忍度：结束延迟允许达到 `5 到 10 分钟`，前提是不要过早结束。
- 标注方式：接受低成本纠错，不接受逐点人工标注。
- 网络条件：默认大多数时候可联网，因此数据导出与训练回收不以纯离线为前提。
- 重构边界：允许对现有实现进行结构性重构。

## 5. 核心设计原则

### 5.1 决策与状态解耦

状态机仍然存在，但其职责从「自行判断开始或结束」收缩为「消费最终决策结果并驱动系统行为」。边界判定的主逻辑迁移到独立决策流水线。

### 5.2 开始与结束必须非对称

开始和结束不是同一个分类问题的两个标签。两者在风险、容忍延迟、观察窗口和保护条件上都不对称，因此必须拆成两个模型与两套触发条件。

### 5.3 模型只给分，不直接生效

任何单个窗口上的模型分数都不得直接触发 `START` 或 `STOP`。最终状态变更必须经过平滑器，平滑器负责处理连续确认、保护期、最短持续时间和迟滞。

### 5.4 先建立数据闭环，再追求模型复杂度

第一阶段的关键不是找最强模型，而是先把样本采集、纠错回灌、训练导出和回放评估这条链路打通。没有稳定数据闭环，复杂模型不会带来稳定收益。

## 6. 总体方案

系统采用「双模型 + 决策平滑 + 人工纠错闭环」方案。

- `StartModel`
  负责回答：当前是否进入了足以开始记录的状态。
- `StopModel`
  负责回答：当前是否进入了足以结束记录的状态。
- `DecisionSmoother`
  负责把连续窗口分数转换为最终 `START / STOP / HOLD` 决策。
- `DecisionFeedbackStore`
  负责记录用户对自动开始 / 自动结束结果的纠错反馈。
- `TrainingSampleExporter`
  负责导出训练样本、模型分数、决策结果和反馈标签。
- `ReplayEvaluator`
  负责基于历史日志回放评估模型与平滑器表现。

### 6.1 决策链路

建议的运行链路如下：

1. `BackgroundTrackingService` 收集定位、步数、加速度、Wi-Fi、地点上下文等原始信号。
2. `SignalSnapshot` 对单次采样做标准化封装。
3. `FeatureWindowAggregator` 基于滚动窗口生成聚合特征。
4. `StartModel` 和 `StopModel` 分别输出 `start_score` 与 `stop_score`。
5. `DecisionSmoother` 结合当前记录状态、保护期和连续确认结果，输出 `START / STOP / HOLD`。
6. 状态机消费最终决策，驱动开始记录、结束记录、通知与前台服务生命周期。
7. 日志与反馈链路记录整次决策过程，用于后续训练与回放。

## 7. 决策契约

### 7.1 输入

每个决策窗口至少包含以下输入：

- 原始聚合特征。
- 当前是否处于记录状态。
- 当前候选状态持续时长。
- 开始保护期剩余时间。
- 当前模型版本与阈值配置版本。

### 7.2 输出

决策引擎的统一输出为：

- `START`
- `STOP`
- `HOLD`

其中：

- `StartModel` 和 `StopModel` 的输出仅为分数，不直接对外暴露状态切换语义。
- `DecisionSmoother` 是唯一有权输出最终状态切换决策的模块。

### 7.3 决策约束

- 仅在当前未记录时评估开始候选。
- 仅在当前记录中时评估结束候选。
- 开始后必须进入保护期，保护期内不得输出 `STOP`。
- 结束候选必须同时满足模型高分和规则层硬约束。

## 8. 数据采集与窗口设计

### 8.1 原始信号

第一阶段模型和规则共用的原始输入至少包括：

- 定位点序列。
- 速度与精度信息。
- 步数变化。
- 加速度摘要。
- Wi-Fi 变化信息。
- 地点上下文，例如常驻地点或停留区域。
- 当前追踪状态及其持续时长。

### 8.2 窗口化策略

采用固定步长滚动窗口：

- 每 `15 秒` 生成一个样本。
- 每个样本聚合最近 `30 秒 / 60 秒 / 180 秒` 三档窗口特征。

每个窗口样本同时生成两个标签位：

- `start_target`
- `stop_target`

不采用单模型三分类结构，避免开始和结束的目标相互干扰。

### 8.3 特征集合

第一阶段特征以聚合统计为主，不直接输入高频原始时序。

- 位移特征：总位移、净位移、位移增长率。
- 速度特征：平均速度、最大速度、速度方差、低速占比。
- 定位质量特征：平均精度、差定位占比、有效定位点数。
- 步行特征：步数增量、步频。
- 加速度特征：均值、方差、峰值次数。
- 停留特征：连续静止时长、小范围徘徊半径。
- 环境特征：Wi-Fi 是否切换、是否位于常驻地点附近。
- 上下文特征：当前是否已在记录、候选状态持续时间、开始保护期剩余时长。

## 9. 标注与反馈闭环

### 9.1 标注策略

不要求用户逐点标注真实轨迹，改用「自动决策后人工纠错」模式。

用户只需对一次自动开始或自动结束给出以下反馈之一：

- 开始太早。
- 开始太晚。
- 结束太早。
- 结束太晚。
- 本次正确。

### 9.2 标签回灌

系统根据用户反馈，将纠错结果映射到相应时间段的窗口样本，自动生成或修正训练标签。

最小完整闭环的第一版先采用更保守的分层策略：

- Android 端只导出原始决策证据，不在手机端硬编码训练标签规则。
- 手机导出格式采用 `jsonl`，每行包含 `eventId`、`timestampMillis`、`finalDecision`、`startScore`、`stopScore`、`features`、`feedbackLabel` 等原始字段。
- WSL 训练脚本先读取手机导出的 `jsonl`，再执行 `feedbackLabel + finalDecision -> start_target / stop_target` 的预处理。
- `START_TOO_LATE` 与 `STOP_TOO_LATE` 在第一版中先跳过，不强行映射为高置信标签。

第一阶段遵循以下策略：

- 正样本只覆盖真实边界附近较窄时间窗。
- 大部分非边界窗口视为负样本。
- 模糊区间不强行赋予高置信标签。

### 9.3 导出内容

第一版手机导出的训练样本至少保留以下字段：

- `eventId`
- `timestampMillis`
- `finalDecision`
- `startScore`
- `stopScore`
- `features`
- `feedbackLabel`

如有需要，允许附带当前内部状态等附加原始字段，但这些附加字段不应改变训练侧对核心字段的依赖关系。

## 10. 模型方案

### 10.1 第一阶段模型

第一阶段采用轻量二分类模型。

- 基线模型：`Logistic Regression`
- 主力候选：`XGBoost`
- 备选模型：`LightGBM`

实施顺序固定为：

1. 先用 `Logistic Regression` 建立基线。
2. 若基线不足，再尝试 `XGBoost`。
3. 若树模型仍不足，再评估更复杂方案。

### 10.2 不采用的方案

第一阶段不采用以下路线：

- 通用 LLM 作为主决策器。
- 直接输入高频原始 IMU 序列的深度神经网络。
- 需要 GPU 参与训练或推理的方案。

## 11. 手机端推理与配置

### 11.1 第一阶段推理方式

第一阶段优先支持 `Logistic Regression` 的原生 Kotlin 推理，不引入额外推理框架。

导出物包括：

- `start_model.json`
- `stop_model.json`
- `feature_config.json`
- `threshold_config.json`
- `decision-model-bundle.json`

对应实现文件与格式约束：

- 训练脚本：`tools/decision-model/train_decision_models.py`
- 回放脚本：`tools/decision-model/replay_decision_run.py`
- 导出契约：`docs/superpowers/specs/decision-model-export-format.md`

其中：

- Android 端优先导入并读取 `decision-model-bundle.json`。
- 若没有导入包，则回退到应用内置默认模型与阈值。
- 保留四个基础 JSON 文件，便于训练、回放与单文件排查。

- 模型文件保存权重、偏置、特征顺序和标准化参数。
- 特征配置保存特征名、顺序、缺失值处理和版本信息。
- 阈值配置保存开始与结束的触发阈值、迟滞阈值和平滑参数。

### 11.2 配置原则

- 模型版本与阈值版本必须可追踪。
- 特征顺序必须稳定，禁止端上和训练侧各自推断。
- 模型升级与阈值升级应允许独立发布。

## 12. 决策平滑与保护机制

模型输出不能直接控制开始或结束，必须经过平滑器。

### 12.1 第一阶段参数方向

- 开始：连续 `2 到 3` 个窗口超过阈值才允许开始。
- 结束：连续 `4 到 8` 个窗口超过阈值才允许结束。
- 开始保护期：`2 到 5 分钟`。
- 结束观察期：`5 到 10 分钟`。

### 12.2 平滑器职责

平滑器至少包含以下机制：

- 连续确认。
- 迟滞阈值。
- 开始后的最短持续时间。
- 结束前的低运动持续时间检查。
- 重新出现移动证据时取消结束候选。

### 12.3 责任边界

- 模型负责表达「像不像开始边界」和「像不像结束边界」。
- 平滑器负责表达「现在能不能真的开始」和「现在能不能真的结束」。
- 任何需要结合状态持续时间、保护期或历史分数序列的约束，都属于平滑器职责。

## 13. 代码重构方案

### 13.1 新增核心模块

建议引入以下结构：

- `tracking/pipeline/SignalSnapshot`
- `tracking/pipeline/FeatureWindowAggregator`
- `tracking/pipeline/FeatureVector`
- `tracking/model/StartDecisionModel`
- `tracking/model/StopDecisionModel`
- `tracking/model/LinearModelRunner`
- `tracking/decision/DecisionSmoother`
- `tracking/decision/TrackingDecisionEngine`
- `tracking/feedback/DecisionFeedbackStore`
- `tracking/export/TrainingSampleExporter`
- `tracking/eval/ReplayEvaluator`

### 13.2 职责调整

`BackgroundTrackingService` 的职责收敛为：

- 收集系统信号。
- 构造 `SignalSnapshot`。
- 驱动 `TrackingDecisionEngine`。
- 响应最终的 `START / STOP / HOLD` 决策。
- 维护前台服务、通知和生命周期。

以下能力应从服务中迁出：

- 特征聚合。
- 模型推理。
- 决策平滑。
- 反馈记录。
- 训练样本导出。
- 历史日志回放评估。

## 14. 训练与评估流程

### 14.1 训练流程

离线训练流程固定为：

1. 从设备导出带反馈的窗口样本。
2. 在 WSL 中使用 Python 工具链训练。
3. 输出模型文件、特征配置和阈值配置。
4. 将模型文件放回 Android 项目供端上推理使用。

第一阶段建议工具链为：

- `scikit-learn`
- `xgboost`
- `pandas`

### 14.2 核心评估指标

禁止仅以普通准确率作为主指标。核心指标为：

- 误开始率。
- 误结束率。
- 开始延迟。
- 结束延迟。

优化优先级固定为：

1. 降低误结束率。
2. 降低误开始率。
3. 优化开始延迟。
4. 优化结束延迟。

### 14.3 回放评估

每次训练后必须执行历史日志回放，至少验证以下内容：

- 模型分数曲线是否稳定。
- 最终触发点是否符合预期。
- 错误主要来自特征、模型还是平滑器。

回放结果必须能追溯到具体日志和具体决策窗口。

## 15. 分阶段落地顺序

### 阶段 1：结构重构

- 抽离信号采集与窗口聚合。
- 建立标准化特征向量定义。
- 搭建日志落盘与导出链路。

### 阶段 2：反馈闭环

- 增加人工纠错入口。
- 保存反馈记录并关联到自动决策事件。

### 阶段 3：基线模型

- 用 `Logistic Regression` 训练 `StartModel` 和 `StopModel`。
- 端上接入原生 Kotlin 推理。
- 接入 `DecisionSmoother`。

### 阶段 4：评估与迭代

- 做回放评估。
- 重点优化误结束与误开始。
- 必要时再切换到 `XGBoost`。

## 16. 验收标准

第一阶段完成时，至少应满足以下标准：

- 能稳定采集窗口样本并导出。
- 能记录用户对开始 / 结束决策的纠错反馈。
- 能在 WSL 中完成 CPU 训练。
- 能在 Android 端加载并执行双模型推理。
- 开始与结束的最终决策由模型分数与平滑器共同决定。
- 可以对历史日志进行回放评估并定位错误来源。

## 17. 风险与应对

- 标签噪声高：通过窄边界正样本和模糊区间降权策略缓解。
- 个体数据不足：先用线性模型做基线，不提前引入复杂模型。
- 模型分数抖动：通过平滑器吸收瞬时波动。
- 重构范围较大：先稳定新流水线，再逐步替换旧逻辑。

## 18. 开放问题

以下问题不影响第一阶段启动，但需要在实现中持续收敛：

- 当前常驻地点与停留区域的定义是否需要统一为同一套地点上下文抽象。
- 当前已有规则中的哪些信号应直接复用，哪些应在重构时废弃。
- 用户反馈入口放在记录结束页、历史页，还是通知交互中，哪种成本最低。
- 第一阶段是否需要为误开始和误结束分别设计不同的人工纠错文案与交互路径。

## 19. 结论

本设计选择「双二分类轻量模型 + 强平滑 + 人工纠错闭环」作为开始 / 结束记录决策的第一阶段方案。该方案不依赖 GPU，不依赖通用 LLM，符合个人使用场景下对可维护性、可训练性和可迭代性的要求。

第一阶段最重要的目标不是立即替换全部现有规则，而是建立一条完整、可观测、可导出、可回放的数据与决策流水线。在此基础上，模型能力才有持续优化的空间。
