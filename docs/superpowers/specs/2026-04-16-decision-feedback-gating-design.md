# 自动开始/结束与待标记列表的硬门槛筛选设计

## 文档定位

- 文档类型：第一阶段补充设计
- 适用范围：自动开始、自动结束、待标记反馈列表的事件准入策略
- 关联设计：
  - [开始 / 结束记录决策模型设计](/Users/a555/StudioProjects/track-record-new/docs/superpowers/specs/2026-04-15-start-stop-decision-model-design.md)
  - [2026-04-15 对话纪要：开始/结束记录模型方案讨论](/Users/a555/StudioProjects/track-record-new/docs/conversations/2026-04-15-start-stop-model-discussion-record.md)

## 1. 背景

当前系统已经具备以下能力：

- 基于窗口特征、双二分类模型和平滑器输出 `START / STOP / HOLD`。
- 将决策事件落库，并在历史页展示最近的可反馈事件。
- 支持用户对自动开始或自动结束进行人工纠错。

但当前事件进入自动决策与反馈列表的准入条件仍然过宽，导致两个问题：

- 室内漂移、弱 GPS、常驻地点附近的小范围活动，可能仍然触发自动开始候选。
- 即使某次自动事件的证据质量较差，也可能进入待标记列表，浪费人工反馈成本。

用户希望采用更保守的策略：宁可少触发、少展示，也不让低质量事件进入自动开始、自动结束和待标记列表。

## 2. 设计目标

### 2.1 目标

- 为自动开始增加保守型硬门槛，降低弱证据触发。
- 为自动结束保留可用性，同时避免把低质量结束事件送入待标记列表。
- 将“是否允许自动决策”与“是否允许进入待标记列表”拆成两层判断。
- 让拒绝原因可观测、可存储、可用于后续调参与回放。

### 2.2 非目标

- 本设计不替换现有双模型与平滑器架构。
- 本设计不引入加权评分或软阈值机制。
- 本设计不一次性重构训练脚本标签映射规则。
- 本设计不改变用户反馈类型定义。

## 3. 用户偏好

本次设计明确采用以下偏好：

- 策略风格：保守。
- 准入方式：硬门槛，而不是加权打分。
- 核心一票否决条件：
  - `GPS 差 / 无信号`
  - `没有步数 / 加速度运动证据`
  - `仍在常驻地点附近`

同时，用户接受自动结束与自动开始采用不完全对称的门槛，以避免结束链路因门槛过硬而长期悬挂。

## 4. 总体方案

系统新增两层门槛：

1. `决策门槛`
   用于决定当前窗口是否允许自动开始或自动结束继续向下游流转。
2. `反馈门槛`
   用于决定某次已经发生的自动事件是否允许进入待标记列表。

设计原则如下：

- 自动开始采用最严格的硬门槛。
- 自动结束采用“保守但不自废武功”的门槛。
- 待标记列表的门槛比自动决策更严格。

## 5. 维度定义

第一阶段统一定义以下门槛维度：

- `gps_quality_pass`
  含义：最近窗口内存在足够数量的有效定位样本，且平均精度、连续有效性达标。
- `motion_evidence_pass`
  含义：最近窗口内存在步数变化、显著运动或加速度波动中的至少一种有效运动证据。
- `frequent_place_clear_pass`
  含义：当前不在常驻地点附近。
- `start_feedback_eligible`
  含义：本次自动开始事件证据足够干净，允许进入待标记列表。
- `stop_feedback_eligible`
  含义：本次自动结束事件证据足够干净，允许进入待标记列表。

同时记录失败原因：

- `gps_missing`
- `gps_poor_accuracy`
- `motion_missing`
- `inside_frequent_place`
- `stop_low_confidence`
- `feedback_blocked_low_quality`

## 6. 自动开始硬门槛

自动开始必须同时满足以下条件：

1. `gps_quality_pass = true`
2. `motion_evidence_pass = true`
3. `frequent_place_clear_pass = true`
4. `StartModel` 分数达到阈值，且满足连续窗口确认

只要任意一项不满足：

- 不允许触发自动开始。
- 不写入可供反馈的自动开始事件。

该策略用于直接阻断以下场景：

- 无 GPS 或 GPS 漂移导致的室内误开始。
- 仅有位置抖动、没有真实运动证据的误开始。
- 仍在家/公司等常驻地点附近的小范围活动误开始。

## 7. 自动结束门槛

自动结束不直接复用自动开始的全部一票否决条件，而采用不对称策略。

自动结束至少需要满足：

1. 已达到最短记录时长。
2. 已满足结束观察期与连续窗口确认。
3. 最近窗口呈现持续静止证据：
   - 低速
   - 低步数
   - 低加速度
   - `StopModel` 分数持续达标

自动结束的特殊规则：

- `仍在常驻地点附近` 不作为结束否决条件，反而可作为辅助支持证据。
- `GPS 差 / 无信号` 不直接禁止结束决策，否则地库、楼内、电梯等场景可能长期结束不掉。

但若自动结束发生在低质量定位条件下，该事件默认不进入待标记列表。

## 8. 待标记列表准入规则

待标记列表的目标不是展示所有自动事件，而是只展示“值得人工反馈”的高质量样本。

### 8.1 自动开始事件进入列表的条件

自动开始事件只有在以下条件全部满足时才进入待标记列表：

- 触发时 `gps_quality_pass = true`
- 触发时 `motion_evidence_pass = true`
- 触发时 `frequent_place_clear_pass = true`
- 本次开始事件不是低可信边界样本

### 8.2 自动结束事件进入列表的条件

自动结束事件只有在以下条件满足时才进入待标记列表：

- 满足结束观察期和最短记录时长
- 结束时的静止证据完整
- 若存在严重 `GPS 差 / 无信号`，则默认不进入列表
- 若结束证据过弱或原因不明，则默认不进入列表

### 8.3 用户可见行为

在历史页中：

- 高质量自动事件：正常进入待标记列表，允许反馈。
- 低质量自动事件：允许系统内部记录，但不出现在待标记列表中。

这样可以确保用户看到的反馈事件数量更少，但质量更高。

## 9. 存储与可观测性

为支持后续回放与调参，决策事件需要补充门槛结果与阻断原因。

建议在决策事件存储中新增：

- `gpsQualityPass`
- `motionEvidencePass`
- `frequentPlaceClearPass`
- `feedbackEligible`
- `feedbackBlockedReason`

如需保持第一阶段实现简洁，也可先将其序列化进现有 `featureJson` 或新增诊断 JSON 字段，但字段语义必须稳定。

## 10. 组件职责

### 10.1 决策门槛评估器

新增一个轻量门槛评估层，位于：

- `FeatureWindowAggregator` 之后
- `TrackingDecisionEngine` 最终输出之前或之后的事件准入层

其职责是：

- 基于当前窗口和上下文计算门槛布尔值
- 为自动开始 / 自动结束 / 列表准入分别产出结论
- 产出阻断原因

### 10.2 历史页反馈列表

历史页不再简单展示“最近的 START / STOP 事件”，而应展示：

- 最近的
- 已通过反馈门槛
- 尚未被隐藏

的自动事件。

## 11. 实现策略

建议按以下顺序落地：

1. 先补充门槛结果模型与阻断原因枚举。
2. 在自动开始链路接入硬门槛。
3. 在自动结束链路接入不对称门槛。
4. 扩展决策事件落库内容。
5. 将历史页列表改为只展示 `feedbackEligible = true` 的事件。
6. 在诊断存储中展示本次门槛结果与阻断原因。

## 12. 测试策略

至少补充以下测试：

- 自动开始在 `gps_missing` 时不得触发。
- 自动开始在 `motion_missing` 时不得触发。
- 自动开始在 `inside_frequent_place` 时不得触发。
- 自动结束在低质量 GPS 条件下可以结束，但不得进入待标记列表。
- 历史页只展示通过反馈门槛的事件。
- 阻断原因能正确落库并回读。

## 13. 风险与权衡

本设计的主要收益是显著提高待标记样本质量，但也带来两个明确权衡：

- 自动开始触发数量会下降。
- 可供人工标记的事件数量也会下降。

这是有意为之，因为当前目标不是尽可能多收样本，而是优先收集“高质量、低歧义”的样本。

## 14. 最终结论

本设计采用“保守 + 硬门槛 + 分层准入”方案：

- 自动开始：严格硬门槛。
- 自动结束：保守但不完全对称。
- 待标记列表：比自动决策更严格。

最终效果应为：

- 低质量自动开始显著减少。
- 低质量自动结束不再进入待标记列表。
- 用户看到的待标记事件更少，但更值得标记。
