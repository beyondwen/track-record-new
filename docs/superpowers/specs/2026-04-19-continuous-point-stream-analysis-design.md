# 连续点流采集与延迟分段分析设计

## 1. 摘要

本设计将自动轨迹系统从「在线开始 / 结束决策」重构为「连续点流采集 + 延迟本地分析」架构。

新方案的核心变化如下：

- 采集层只负责稳定、低功耗地持续记录位置点，不再在采集期决定「开始记录」或「结束记录」。
- 分析层基于连续点流，在本地延迟识别静止段、动态段和停留点。
- 数据模型以原始点流表为事实源，分析结果表为长期保留的衍生层。
- 第一优先级是避免将静止误判为动态，允许短距离真实移动被保守处理。
- 旧的 `auto_track_session` 中心模型不保留兼容路径，直接由新模型替代。

## 2. 背景与问题定义

当前项目已经具备后台追踪能力，Manifest 中已经声明后台定位、前台定位服务、活动识别、开机恢复和忽略电池优化等权限，并由 [`BackgroundTrackingService.kt`](/Users/a555/StudioProjects/track-record-new/app/src/main/java/com/wenhao/record/tracking/BackgroundTrackingService.kt) 负责运行期状态切换与采样调度。

现有实现存在以下结构性问题：

- 采集层和边界决策层耦合过深，服务本身既要采点，又要判断什么时候开始、什么时候结束。
- 数据模型以 `auto_track_session` 为中心，天然偏向「在线切段」，不适合后续重放与算法重跑。
- 为了降低误判，在线状态机会不断堆叠条件，复杂度持续上升，但仍然难以解决楼内漂移、短暂停车、弱信号抖动等问题。
- 用户当前更关心不要把静止误判成出行，而不是尽可能早地产生一段轨迹。

因此，本次设计不再继续优化「在线开始 / 结束」，而是直接转换问题定义：

- 采集期只回答「现在要不要继续记点，以及用什么频率记点」。
- 分析期才回答「哪些点属于静止段，哪些点属于动态段」。

## 3. 设计目标

### 3.1 目标

- 建立连续点流事实源，支持持续采点、延迟分析和窗口重跑。
- 在本地延迟识别静止段、动态段和停留点，不依赖服务端分析。
- 在准确率与续航之间取得平衡，采用分层采样而非全天高频 GPS 常驻。
- 以「避免静止误判为动态」作为第一优先级，允许边界存在一定延迟。
- 将原始点短期保留，分析结果长期保留，控制存储体积与隐私风险。

### 3.2 非目标

- 不保留旧 `auto_track_session` 的兼容读写路径。
- 不在采集期输出最终的开始 / 结束边界。
- 不在第一阶段引入服务端分析或在线模型训练。
- 不追求最短边界延迟。
- 不在第一阶段解决所有活动类型分类问题，聚焦静止 / 动态二分与停留点识别。

## 4. 用户偏好与约束

- 最不能接受的问题：将静止误判成动态。
- 对准确率与续航的取舍：两者都要考虑，但允许方案偏保守。
- 采样偏好：采用分层采样，而不是全时均匀高频采样。
- 升频条件：运动识别与位置位移 / 速度双信号同时满足。
- 降频条件：连续 `5 到 10` 分钟低速度、低位移，且运动识别回到 `STILL`。
- 原始点保留策略：短保留，目标区间为 `7 到 14` 天。
- 分析执行位置：全部在本地完成。
- 结果产出时机：延迟产出，不要求边采边给出最终结论。

## 5. 核心设计原则

### 5.1 采集与分析彻底解耦

采集层不再理解「轨迹会话」概念，只负责持续落点。分析层独立消费连续点流，并输出可被 UI 使用的片段和停留结果。

### 5.2 原始点流是真实事实源

任何静止段、动态段、停留点都属于衍生结果，而不是采集时写死的真相。只要原始点仍在保留期内，算法升级后就能重算结果。

### 5.3 分段结果允许延迟，不允许轻率

当证据不足时，优先输出「静止」或「不确定」，而不是急于生成动态段。第一阶段要先压低误判，再逐步找回漏判。

### 5.4 续航优化依赖采样调度，不依赖停止采点

系统不采用「完全停采直到下一次开始」的模式，而是通过低频哨兵采样、高频活跃采样和保守回落来兼顾续航与连续性。

## 6. 总体架构

系统由两层组成：

- `采集层`
  负责运行后台服务、接收定位与活动识别信号、根据当前采样档位调整请求频率，并将所有点追加到原始点流表。
- `分析层`
  负责清洗点流、打分、切段、合并、停留点聚类，并将结果写入分析结果表。

### 6.1 数据流

1. `BackgroundTrackingService` 以低频哨兵模式持续运行。
2. 服务监听定位、活动识别、Wi-Fi 变化等信号，动态切换采样档位。
3. 采集到的原始点统一写入 `raw_location_point`。
4. 当满足分析触发条件时，本地分析器读取尚未分析的点区间。
5. 分析器完成点清洗、静止 / 动态识别、停留点聚类，并写入 `analysis_segment` 与 `stay_cluster`。
6. UI、历史记录、地图页统一读取分析结果，而不是读取服务内部状态。

### 6.2 运行边界

- 采集层不输出 `START` / `STOP` 事件。
- 分析层不控制定位服务启停，只负责产出结果。
- 结果页面默认展示分析层的稳定结果，而不是瞬时状态。

## 7. 数据模型设计

### 7.1 原始点流表

新增 `raw_location_point`，作为唯一事实源。建议字段如下：

- `pointId`
- `timestampMillis`
- `latitude`
- `longitude`
- `accuracyMeters`
- `altitudeMeters`
- `speedMetersPerSecond`
- `bearingDegrees`
- `provider`
- `sourceType`
- `isMock`
- `wifiFingerprintDigest`
- `activityType`
- `activityConfidence`
- `samplingTier`

说明：

- 该表仅追加写入，不做在线分段归属。
- `samplingTier` 用于区分哨兵档、疑似移动档、活跃档，便于后续分析和调试。
- `wifiFingerprintDigest` 只保留摘要，不保存完整可识别的 Wi-Fi 明文列表。

### 7.2 分段结果表

新增 `analysis_segment`，表示分析器输出的稳定片段。建议字段如下：

- `segmentId`
- `startPointId`
- `endPointId`
- `startTimestamp`
- `endTimestamp`
- `segmentType`
- `confidence`
- `distanceMeters`
- `durationMillis`
- `avgSpeedMetersPerSecond`
- `maxSpeedMetersPerSecond`
- `analysisVersion`

其中 `segmentType` 仅允许：

- `STATIC`
- `DYNAMIC`

### 7.3 停留点结果表

新增 `stay_cluster`，表示停留点聚类结果。建议字段如下：

- `stayId`
- `segmentId`
- `centerLat`
- `centerLng`
- `radiusMeters`
- `arrivalTime`
- `departureTime`
- `confidence`
- `analysisVersion`

### 7.4 可重建投影视图

面向 UI 的「一次出行」视图、按天历史聚合、地图展示缓存都视为投影视图或派生缓存，不作为事实源。它们可以来自 `analysis_segment` 与 `stay_cluster` 的二次组装。

## 8. 采样与分析触发策略

### 8.1 采样档位

系统采用三档采样：

- `哨兵档`
  默认档位。低频采点，用于保证连续性和后续可重放性。
- `疑似移动档`
  当系统捕获到可能开始移动的证据时临时升频，用于缩短动态段边界的不确定区。
- `活跃档`
  当连续信号证明确实处于移动中时进入高频采样。

### 8.2 升频条件

从低频升到高频必须满足双信号条件：

- 活动识别出现持续的 `WALKING`、`ON_BICYCLE` 或 `IN_VEHICLE` 迹象。
- 位移或速度信号达到可疑阈值，而非单点抖动。

只有两类证据同时出现时，系统才允许升到更高频率，以降低静止漂移误判。

### 8.3 降频条件

系统采用保守回落：

- 连续 `5 到 10` 分钟低速度、低位移。
- 同期活动识别回到 `STILL`。
- 当前位置重新落入稳定停留区或没有继续扩大的位移趋势。

只有满足以上条件，活跃采样才回落到哨兵档。

### 8.4 分析触发

分析任务采用延迟触发，不边采边做最终切段。建议触发时机如下：

- 从活跃档回落到哨兵档时，提交一次分析任务。
- 长时间连续移动时，按固定增量窗口触发中间分析，避免单次分析窗口过大。
- 服务重启或系统恢复后，优先补算尚未分析的点区间。

### 8.5 分析窗口

分析器以「未分析连续点区间」为单位工作，每次分析前后附带重叠缓冲区，用于后续边界修正。分析结果只提交中间稳定区域，窗口边缘留给下一次分析继续修正。

## 9. 算法流水线

分析器建议拆为以下 5 个步骤。

### 9.1 点清洗

对原始点进行保守清洗，去除以下明显异常：

- 时间倒退点
- 重复点
- 精度极差点
- 不可能速度点
- 短时跳点

第一阶段只做保守剔除，不尝试复杂插值修复。

### 9.2 点状态打分

以点或小窗口为粒度，分别计算：

- `static_score`
- `dynamic_score`

特征来源包括：

- 位移增量
- 速度与推断速度
- 持续时长
- 定位精度
- Wi-Fi 指纹稳定度
- 活动识别结果与置信度

### 9.3 初步切段

基于分数和硬门槛，将连续点流初步划分为：

- 明显静止段
- 明显动态段
- 不确定段

第一阶段允许保守输出不确定段，不强行归类。

### 9.4 段合并与边界修正

对初步切段结果进行后处理：

- 合并被短噪声切碎的动态段
- 合并同一停留区域内被漂移打断的静止段
- 将过短、证据不足的动态段回收为静止扰动
- 对边界进行回溯修正，使起点和终点更接近真实出行边界

### 9.5 停留点聚类

在稳定静止段之上继续聚类，生成停留点结果。停留点聚类输出用于 UI 展示、历史摘要和后续常驻地点建模。

## 10. 保护规则与阈值策略

### 10.1 动态段成立条件

动态段至少需要满足一组强证据，例如：

- 连续多点累计位移达到阈值，且持续时间足够
- 速度持续高于低速门槛，而不是单点尖峰
- 活动识别持续显示移动态
- Wi-Fi 指纹明显切换，且伴随位置变化

### 10.2 优先判静止的场景

以下场景默认优先判为静止或不确定：

- 只有少数跳点，且定位精度差
- 总位移很小，但点在小半径范围内漂移
- 短暂停留后很快回到原区域
- 楼内、电梯、地下车库等弱信号场景
- 等红灯、堵车、短暂停车等短时低速片段

### 10.3 阈值层次

阈值不采用单一打死的数值，而采用分层策略：

- `硬门槛`
  用于直接拦截明显误判，例如最小动态持续时长、最小净位移、最大可接受精度。
- `软分数`
  让速度、位移增长、Wi-Fi 变化、活动识别共同贡献 `dynamic_score`。
- `回收规则`
  即使某段初步判成动态，只要总距离太短、持续太短，或边界两侧都落在同一停留区，就回收为静止扰动。

### 10.4 第一阶段取向

第一阶段明确偏保守：

- 小区内活动、楼内走动、下楼拿快递等短活动，不追求全部识别为动态段。
- 真正的步行、骑行、驾车出行允许延迟几分钟再稳定产出。

## 11. 存储与保留策略

### 11.1 保留周期

- `raw_location_point` 保留 `7 到 14` 天。
- `analysis_segment` 与 `stay_cluster` 长期保留。

### 11.2 清理策略

- 原始点清理按时间窗口执行，不允许破坏尚未分析完成的点区间。
- 清理前需要确认分析结果已经落库，且相关窗口已标记为完成。

### 11.3 隐私约束

- 第一阶段不上传原始点流到服务端。
- Wi-Fi 相关信息只保留摘要，不保存可直接识别用户环境的明文详情。
- 所有分析结果默认保存在本地数据库。

## 12. 重构边界

本次方案不做旧模型兼容，按新架构直接重建：

- 旧 `auto_track_session` 不再作为主事实源。
- `BackgroundTrackingService` 的职责缩减为采样调度与原始点采集。
- 旧的在线开始 / 结束状态机、实时会话驱动逻辑和相关决策事件存储，需要逐步下线或改造成采样调试用途。
- UI 读取路径统一改为消费分析结果表。

## 13. 验证与测试方案

### 13.1 单元测试

覆盖以下纯算法模块：

- 点清洗
- 点状态打分
- 初步切段
- 段合并与回收
- 停留点聚类

### 13.2 回放测试

使用历史原始点流进行离线回放，重点验证：

- 静止误判为动态的比例
- 动态段是否被过度切碎
- 分析版本升级后结果是否稳定

### 13.3 真机场景测试

至少覆盖以下场景：

- 家中静止半天
- 小区内短距离活动
- 下楼拿快递再返回
- 步行通勤
- 驾车通勤
- 堵车与等红灯
- 地下车库或弱信号场景
- 开机恢复、服务被杀后的恢复

### 13.4 续航与数据量测试

重点观测：

- 24 小时额外耗电
- 每日原始点增长量
- 分析任务单次耗时
- 清理策略执行后的数据库体积变化

### 13.5 第一阶段成功标准

- 静止误判成动态的情况明显少于旧方案。
- 日常通勤能够稳定识别为动态段。
- 全天额外耗电保持在可接受范围内。
- 原始点短保留策略足以控制数据库增长。

## 14. 风险与后续演进

### 14.1 已知风险

- 第一阶段偏保守，可能漏掉极短的真实移动。
- 全本地分析要求分析器必须足够轻量，否则会影响后台稳定性。
- 原始点短保留会限制长期回放与调参窗口。

### 14.2 后续演进方向

- 在本地结果稳定后，再考虑可选的匿名样本导出用于调参。
- 在停留点结果稳定后，再引入常驻地点识别、地点语义命名等上层能力。
- 在静止 / 动态分段稳定后，再决定是否需要额外的轨迹级摘要模型。
