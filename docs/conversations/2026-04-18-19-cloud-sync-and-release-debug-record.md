## 3. 模型与标注方案的关键结论

### 3.1 关于端侧小模型自动决策

用户提出，希望在手机端放一个自训练小模型，根据一组传感器状态决定是否开始记录和停止记录。

围绕这个方向，最终形成的结论是：

- 技术上可行，但现阶段不应该直接把模型放到前台决策链路中。
- 当前最缺的不是模型推理能力，而是高质量标签与稳定的数据闭环。
- 因此，第一阶段应该优先建立「可靠样本采集和标注」机制，而不是直接追求自动决策。

### 3.2 标注策略的收敛

对话中逐步把标注策略收敛为：

- 用户手动点击开始记录到手动点击结束记录之间的数据，视为正样本。
- 其他时段不直接当作绝对负样本，而是更谨慎地构造负样本和边界样本。
- 为了减少噪声，最终采用了更保守的方案：先以手动边界作为可信标签来源。

进一步确认后，又形成了两个重要结论：

- 现阶段先不要让系统自动开始或自动结束记录。
- 在模型训练成熟之前，自动记录能力应先下线，避免继续产生噪声标签。

## 4. 自动记录策略调整

基于上述判断，本次会话中已明确：

- 自动记录相关能力先移除或停用。
- 当前以「用户手动开始 / 手动结束」为唯一可信记录边界。
- 后续训练完成后，再由模型接管是否开始记录和停止记录的判断。

这一步的意义在于：

- 先把标签质量提上来。
- 避免继续把错误自动判断写回训练集。
- 让模型训练从「脏数据闭环」切换到「干净人工边界闭环」。

## 5. GitHub Actions 与发布链路排障过程

### 5.1 签名与 Secret 配置问题

在 GitHub Actions 打包阶段，用户遇到了与签名和 Secret 配置有关的多类问题，包括：

- `security` 项配置后，Token 没有被打包进 APK。
- Secret 名以 `GITHUB_` 开头时，GitHub 报错不允许创建。
- 工作流表达式中直接使用 `secrets.RELEASE_KEYSTORE_BASE64 != ''` 触发：
  - `Unrecognized named-value: 'secrets'`

围绕这些问题，逐步形成的处理结论是：

- 运行时配置不应再打进 APK，尤其是 Mapbox Token 这类客户端可输入配置。
- 签名相关变量继续保留在 Actions Secret 中。
- Mapbox Token 改为在 App 客户端输入并保存在本机。
- 工作流中的 Secret 判空逻辑需要调整写法，避免在不支持的上下文直接使用 `secrets` 表达式。

## 6. App 更新能力相关问题

### 6.1 检查更新失败

用户多次反馈 App 中点击「检查更新」提示失败。排查过程中确认过以下因素：

- 手机代理会影响更新请求访问 GitHub 或相关资源。
- 配置错误或版本未递增会让检查更新看起来像是失败。
- 某些版本中关于页按钮还出现过显示异常，影响用户判断。

最终结论是：

- 网络代理会直接影响更新检查结果。
- 版本号必须递增，才能稳定验证更新链路。

### 6.2 点击更新后闪退或无反应

用户还反馈：

- App 已识别到有新版本，但点击「更新」后闪退。
- 某些版本中点击更新后没有明显反应。

在这一阶段，围绕下载、安装器调起和未知来源安装权限做了补强和多轮验证。后续用户也通过 GitHub 下载最新版 APK 进行手工安装验证。

最终形成的实际使用结论是：

- 只要 APK 已正确安装，App 内更新检查本身可以继续保留。
- 若某次内置更新安装链路异常，仍可通过 GitHub Release 手工下载安装作为兜底路径。

## 7. 关于 Mapbox Token 与仓库公开性的讨论

围绕是否可以将项目切换为 Public，以及 APK 是否会被反编译，形成了如下结论：

- Android APK 无法做到绝对不可反编译，只能提升逆向成本。
- 如果 Mapbox Token 直接打进 APK，就存在被提取的风险。
- 因此更稳妥的方式是：
  - Mapbox Token 不写入仓库
  - 不打进 APK
  - 改为用户在客户端输入并仅保存在当前设备

对于仓库公开与 GitHub Actions 计费，也完成了概念澄清：

- Public 与 Private 都可能消耗 GitHub Actions 配额，只是计费策略和免费额度不同。
- 是否收费取决于 Runner 类型、运行时长与账户套餐，而不是单纯由仓库公开与否决定。

## 8. 训练样本上云链路的设计与落地

### 8.1 为什么不让客户端直连公网 MySQL

用户提出是否可以直接从客户端连接公网 MySQL，或者由客户端自己填写数据库信息。

最终给出的明确建议是：

- 不推荐 App 直连公网 MySQL。
- 不推荐把数据库账号、密码和地址长期暴露给客户端。
- 更合理的路径是由 App 调用一个轻量中转层，再由中转层写数据库。

在用户明确表示没有额外中转服务部署空间后，最终选定方案为：

- `App -> Cloudflare Worker -> Hyperdrive -> MySQL`

这个方案的优点是：

- 不需要自建常驻服务器。
- 客户端不持有数据库凭据。
- 可通过 Token 做简单鉴权。
- 服务端逻辑集中，便于后续扩展。

### 8.2 Cloudflare Worker 接入过程

在 Worker 实际接入过程中，用户完成了以下操作：

- 使用 `wrangler` 进行本地登录。
- 由于浏览器回跳 `localhost` 失败，改用 API Token 的方式登录。
- 配置 Hyperdrive 绑定。
- 配置 `UPLOAD_TOKEN` Secret。
- 成功部署 Worker。

部署成功后，线上 Worker 地址形如：

- `https://track-record-worker.beyondlenovo.workers.dev`

### 8.3 样本上传链路中的关键故障与根因

训练样本上传能力打通过程中，先后遇到了多层问题。按真实定位顺序整理如下。

#### 第一层：本地导出内存失败

在导出较长时间范围的手动样本时，出现：

- `CursorWindow NO_MEMORY`

根因是：

- 一次性导出长时间窗口样本，导致本地游标窗口内存压力过大。

对应修复提交：

- `63c6ba3 fix(样本上传): 分页导出长时段手动样本`

#### 第二层：上传包过大与缺少批次诊断

在解决本地导出问题后，又确认：

- 上传请求仍可能因为一次性数据过大而失败。
- 客户端缺少对失败批次的日志与明确提示。

对应修复提交：

- `da248d3 fix(样本上传): 分批上传样本并补充日志`

#### 第三层：Worker 可达性不透明

在网络、配置和服务端排障阶段，用户无法直接区分：

- Worker 不可达
- 路径错误
- 鉴权失败
- 服务端 500

因此新增了客户端诊断能力。对应提交：

- `6aaf39f feat(网络诊断): 增加 Worker 连通性测试按钮`

这一步之后，关于页可以直接区分：

- `405`：Worker 可达，路径存在但方法不对
- `401/403`：Worker 可达，但鉴权失败
- `404`：Worker 可达，但路径配置错误
- `5xx`：Worker 可达，但服务端内部异常
- `IOException`：Worker 无法连接

#### 第四层：服务端错误被吞掉

当 Worker 真的返回 500 时，客户端最开始只能看到泛化错误，难以继续定位。

因此补了 Worker 真实错误透传。对应提交：

- `6573258 fix(worker): 返回训练样本入库错误详情`

#### 第五层：Hyperdrive 不支持预处理协议

在真实错误透传之后，最终定位到根因：

- Hyperdrive 不支持 `COM_STMT_PREPARE`

而 Worker 中使用的 `mysql2` `execute(...)` 默认会走预处理协议，因此写库失败。

最终修复方式是：

- 将 `connection.execute(...)` 改为 `connection.query(...)`

对应提交：

- `b162fd5 fix(worker): 改用 query 兼容 Hyperdrive`

### 8.4 样本上传链路最终状态

在完成上述修复并由用户重新 `wrangler deploy` 之后，链路已确认稳定：

## 10. 历史轨迹上云同步的扩展

### 10.1 扩展动机

在训练样本上云稳定后，用户进一步提出：

- 既然训练样本已经能写数据库，历史轨迹是否也能同步到数据库中。

对此给出的建议是：

- 可以，并且很适合复用现有 Worker 地址与上传 Token。
- 与训练样本不同，历史轨迹上云不应在成功后删除本地历史。
- 历史轨迹更适合作为云端追加同步。

### 10.2 设计决策

围绕历史轨迹上传，最终确定如下设计：

- App 端新增「上传未上传历史轨迹」按钮。
- 复用当前关于页中的 Worker 地址和 Token 配置。
- 服务端新增 `/histories/batch` 路由。
- 数据库新增 `uploaded_histories` 表。
- 用 `(device_id, history_id)` 作为唯一键去重。
- 轨迹点数据直接随记录一起上传，存入 `points_json`。
- 历史轨迹上传成功后只标记为已上传，不删除本地历史。
