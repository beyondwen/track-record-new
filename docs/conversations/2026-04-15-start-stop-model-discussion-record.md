## 2. 对话背景

本次讨论建立在对当前项目结构和核心逻辑完成初步阅读之后。已确认本项目并非简单的 `HelloWorld` 应用，而是一个以轨迹记录为核心的 Android 应用 `TrackRecord`，其当前能力包括：

- 后台自动识别出行并记录轨迹。
- 历史轨迹存储、浏览与地图展示。
- 气压 / 海拔信息展示。
- 基于权限、定位、运动传感器和状态机的后台自动追踪。

在此基础上，用户提出希望进一步增强「自动开启记录」和「自动结束记录」的判断能力。

## 3. 用户核心诉求

用户希望：

- 不再依赖人为手动开始或结束记录。
- 让系统在后台自动决定何时开始记录、何时结束记录。
- 优先减少误开始和误结束。
- 可以接受判断偏慢一些。
- 如果需要，可以接受较大规模代码重构。
- 长期来看，希望形成一个能持续优化的个人使用方案。

## 4. 讨论过程整理

### 4.1 初始方向：是否能在 App 中集成端侧大模型

用户最早提出，希望把 `Gemma` 相关模型预先下载到手机上，再在 App 内利用模型打分，帮助决定开始或结束记录。

围绕这个问题，讨论得出的结论是：

- 技术上可以把模型下载到设备本地，并在 Android 应用内做端侧推理。
- 但通用大模型直接承担传感器行为决策，通常不是最优工程方案。
- 对当前项目来说，更实际的方式不是让大模型持续主导判断，而是先评估其是否适合作为边界裁决器。

### 4.2 追问：后台能否调用大模型

用户随后关注后台调用可行性，特别是：

- 如果模型是系统侧能力，如 `Gemini Nano` / `ML Kit GenAI`，后台是否可用。
- 如果模型是应用自行集成的本地模型，后台是否可用。

得出的判断是：

- `Gemini Nano` / `ML Kit GenAI` 一类系统模型并不适合当前需求，因为后台使用存在明确限制。
- 自行集成的本地模型理论上可以在后台进程中使用，但并不代表适合持续高频后台推理。
- 对当前需求而言，「后台持续运行通用 LLM」在功耗、内存和稳定性上都不划算。

因此，虽然「后台调用模型」并非绝对不可行，但不应作为第一阶段的主路径。

### 4.3 备选方向：使用在线模型

在端侧模型之外，用户又提出使用在线模型的可能性，例如在 App 内配置 API key，将行为特征提交给在线模型，由模型返回开始或结束判断。

围绕在线模型方案，形成了如下判断：

- 技术上完全可行。
- 直接把真实 API key 放进 APK 中并不安全。
- 更合理的做法应是 `App -> 自有后端 -> 模型 API`。
- 在线模型更适合做原型、边界复核或决策解释，不适合作为长期的一线高频判定器。

同时也提出了一个重要工程建议：

- 不应把整段原始轨迹全文直接发送给模型。
- 更推荐发送结构化特征，例如最近一段时间的位移、速度、步数、加速度、定位精度和环境变化等。
- 让模型只返回结构化决策结果，如 `start / stop / hold` 以及置信度。

### 4.4 继续收敛：手机端是否存在更适合的模型

用户随后问到，是否有比 `Gemma` 更适合在手机端使用的大模型。

此处给出的核心判断是：

- 如果只是为了「开始 / 结束记录」这类行为边界判断，通用 LLM 并不是最优解。
- 即便从手机端模型中选，也更应关注：
  - 结构化输出稳定性。
  - 模型体积和内存占用。
  - 持续后台运行的成本。

讨论中提到了一些可能的模型方向，例如 `Qwen2.5-1.5B-Instruct`、`Gemma 3 1B IT`、`Gemma 3n E2B` 等，但最终结论并不是「从这些模型中二选一」，而是：

- 对当前任务，最适合的方向并非通用大语言模型。

### 4.5 关键转折：是否存在更适合做行为检测的模型类型

用户进一步追问：「那种模型对这种行为检测更好？」

在这一阶段，讨论重心从「通用 LLM」切换到「行为识别模型」。

得出的判断是：

- 对于基于定位、步数、加速度等信号做行为边界识别，更适合的不是通用 LLM。
- 更合适的范式包括：
  - `Android Activity Recognition Transition API`
  - 轻量特征模型，如 `Logistic Regression`、`XGBoost`、`LightGBM`
  - 时序行为识别模型，如 `1D CNN`、`TCN`、`GRU`、`LSTM`

同时明确指出：

- 当前需求不是泛化的「识别走路 / 静止 / 乘车」。
- 当前需求是两个最终决策：
  - 是否开始记录
  - 是否结束记录

这使问题定义从活动识别转向了边界决策。

### 4.6 决策偏好澄清

为了让方案更明确，进一步向用户确认了最不能接受的错误类型。用户的回答是：

- 不追求最快响应。
- 希望同时尽量减少误开始和误结束。
- 宁可慢一点判断，也要尽量稳。

在此基础上，又继续确认了网络条件。用户明确表示：

- 手机大部分时候有网。
- 不必强行把「完全离线可用」作为第一阶段硬约束。

这一步的重要意义在于：

- 第一阶段设计不必为了离线闭环牺牲实现效率。
- 可以优先做更容易采集、导出和迭代的数据链路。

### 4.7 三种方向的对比

在需求收敛后，正式提出了三种方向供选择：

1. 直接用在线大模型做边界裁决。
2. 现有状态机初筛，在线大模型复核。
3. 专用行为分类模型替代 LLM。

从工程视角出发，最开始推荐的是方案 2，因为它最稳妥，也最容易接入现有系统。

但用户进一步明确表示：

- 希望做方案 3。
- 即，自建专用模型，而不是依赖在线大模型长期充当决策者。

这成为本次讨论最重要的分叉点。

### 4.8 方案 3 可行性确认

用户随后确认，是否可以由我配合完成方案 3。

对此给出的明确判断是：

- 可以，而且推荐从轻量模型开始，不必一步到位上复杂神经网络。
- 由于这是个人使用场景，数据规模初期较小，更适合先用简单、可解释、可快速验证的模型。

由此引出了后续关于训练成本、标注方式和模型选型的进一步细化。

### 4.9 训练成本讨论：是否需要昂贵 GPU

用户担心方案 3 是否需要昂贵的 GPU 资源。

结论是：

- 第一阶段不需要 GPU。
- 只要先走轻量特征模型路线，CPU 就足够完成训练。
- 在 WSL 环境中即可完成第一阶段训练。

也明确指出，只有在以下情况下才可能需要 GPU：

- 使用高频原始 IMU 时序作为输入。
- 样本量显著增大。
- 引入较复杂的深度神经网络。
- 进行大规模超参数搜索。

但这些都不属于当前第一阶段目标。

### 4.10 标签获取方式讨论：人工逐点标注是否必要

在确认用户接受「人工标注 + 轻量模型」之后，继续讨论了标注方式。

为了降低标注负担，提出并确认了以下策略：

- 不采用逐点手工标注。
- 改为「自动决策后人工纠错」。
- 用户只需在事后反馈：
  - 开始太早
  - 开始太晚
  - 结束太早
  - 结束太晚
  - 本次正确

用户确认可以接受这种方式。

这一步很重要，因为它决定了后续训练数据的获得方式，也决定了系统必须内建反馈存储与样本回灌机制。

### 4.11 延迟容忍度确认

为了确定平滑策略和停止规则，又进一步确认了延迟偏好。

用户选择了最保守的一档，即：

- 只要准确，结束延迟 `5 到 10 分钟` 也可以接受。

这一点直接影响了后续设计中关于：

- 结束观察期。
- 连续确认窗口数量。
- 开始保护期。
- 误结束优先级。

### 4.12 方案设计阶段：双模型与平滑器

在完成需求澄清后，正式进入设计阶段，并逐步向用户展示了完整方案。

设计的核心结论是：

- 不用单个三分类模型。
- 使用两个独立二分类模型：
  - `StartModel`
  - `StopModel`
- 通过平滑器完成最终决策，而不是让模型单次输出直接生效。

同时明确：

- `StartModel` 关注较短窗口。
- `StopModel` 关注更长窗口。
- 开始和结束的逻辑不对称，因此不应强行共享为一个简单分类头。

用户接受了这一总体设计方向。

### 4.13 数据与标注设计阶段

随后进入训练数据设计。

确定的关键点包括：

- 使用固定步长滚动窗口生成训练样本。
- 每 `15 秒` 生成一个样本。
- 每个样本聚合最近 `30 秒 / 60 秒 / 180 秒` 的特征。
- 每个样本同时拥有：
  - `start_target`
  - `stop_target`

第一阶段特征以聚合统计为主，包括：

- 位移类特征。
- 速度类特征。
- 定位质量特征。
- 步行 / 步数特征。
- 加速度摘要特征。
- 停留特征。
- Wi-Fi / 地点上下文特征。
- 当前状态上下文特征。

这部分设计的目标是：

- 避免一开始就引入高频原始时序建模复杂度。
- 先用稳定的聚合特征完成基线模型。

### 4.14 模型训练与手机端集成设计

接着进入模型形态与端上推理的设计阶段。

最终明确的阶段性策略为：

- 第一阶段基线：`Logistic Regression`
- 主力候选：`XGBoost`
- 备选：`LightGBM`

训练顺序固定为：

1. 先用 `Logistic Regression` 建立基线。
2. 若效果不足，再尝试 `XGBoost`。
3. 若仍不足，再评估更复杂方案。

同时，为降低端上推理复杂度，还确定了：

- 第一阶段优先支持 `Logistic Regression` 的原生 Kotlin 推理。
- 导出物采用 JSON 配置形式。
- Android 端直接加载权重、偏置、特征顺序和标准化参数执行推理。

这意味着：

- 第一阶段不依赖额外推理框架。
- 第一阶段模型部署可以非常轻量。

### 4.15 决策平滑与评估指标设计

在模型本体之外，又进一步补充了决策平滑层。

第一阶段的参数方向被确定为：

- 开始：连续 `2 到 3` 个窗口高分才允许开始。
- 结束：连续 `4 到 8` 个窗口高分才允许结束。
- 开始保护期：`2 到 5 分钟`。
- 结束观察期：`5 到 10 分钟`。

并明确提出，系统的主要评估指标不应只看普通准确率，而应聚焦：

- 误开始率。
- 误结束率。
- 开始延迟。
- 结束延迟。

优先级固定为：

1. 先降低误结束率。
2. 再降低误开始率。
3. 再优化开始延迟。
4. 最后再优化结束延迟。

### 4.16 允许重构后的结构设计

用户明确表示「可以进行代码重构，不用考虑风险之类的」，因此设计不再受当前 `BackgroundTrackingService` 结构限制。

在这种前提下，提出了新的模块化结构，包括：

- `SignalSnapshot`
- `FeatureWindowAggregator`
- `FeatureVector`
- `StartDecisionModel`
- `StopDecisionModel`
- `LinearModelRunner`
- `DecisionSmoother`
- `TrackingDecisionEngine`
- `DecisionFeedbackStore`
- `TrainingSampleExporter`

新的职责划分是：

- 特征聚合、模型推理、平滑规则、反馈记录、训练导出、回放评估都迁移到独立模块中。

## 5. 本次对话形成的最终结论

### 5.1 功能方向

不采用通用大语言模型作为开始 / 结束记录的主决策器，而是采用专用轻量模型方案。

### 5.2 第一阶段技术路线

第一阶段采用：

- 双二分类模型。
- 轻量特征输入。
- `Logistic Regression` 作为基线。
- Kotlin 原生端侧推理。
- 强平滑与保护机制。

### 5.3 数据策略

采用「自动决策后人工纠错」的低成本反馈方式，而不是逐点人工标注。

### 5.4 训练资源

第一阶段不需要 GPU，CPU 即可训练。

### 5.5 工程结构

允许并建议对当前追踪服务做模块化重构，以支持：

- 数据采样
- 训练导出
- 模型推理
- 决策平滑
- 回放评估

### 5.6 优先优化目标

优先减少误结束和误开始，而不是追求最短延迟。

### 9.2 按计划推进实现

在正式进入开发后，又补写了实现计划文档，并据此按任务推进落地。

首先完成了任务 2，对应模型运行时、决策平滑器和决策引擎的基础设施。这部分把设计稿中的双二分类模型、线性推理、平滑规则与最终 `start / stop / hold` 决策真正落到了 Kotlin 代码中。

随后完成了任务 3，为 `decision_event`、反馈记录和训练样本导出建立持久化层。期间遇到两个实现层面的具体问题：

在任务 4 中，新建了 `DecisionRuntimeCoordinator`，并把特征聚合、模型打分、平滑决策、事件落库和现有 `BackgroundTrackingService` 串接起来。同时把模型分数等诊断信息接入了调试面板，保证新链路在第一阶段仍然具备可观测性。

### 9.3 历史页方案修正与工具链补齐

任务 5 原计划是直接在历史页为既有记录增加人工纠错按钮，但实际接入时发现一个关键前提并不成立：当前历史页的数据模型并没有直接暴露可用于反馈提交的真实 `decision_event` 标识。

因此，这一阶段没有生硬把反馈按钮挂到现有记录模型上，而是先修正实现方向：

- 先补齐最近 `decision_event` 的真实查询能力。

这次修正很重要，因为它说明设计稿中的闭环方向是正确的，但页面接入层还需要先补一层真实的数据映射，不能直接跳过事件实体。

### 9.6 最小完整闭环补充（2026-04-16）

在上述基础能力已经具备之后，用户进一步提出了一个更明确的落地目标：优先打通「手机采样导出 -> WSL 训练 -> 手机导入模型 -> 后台直接使用新模型」的最小完整闭环，而不是继续扩展更复杂的训练策略。

围绕这个目标，双方最终确认了以下执行约束：

- 手机端在历史页 Hero 区域新增两个动作：`导出训练样本` 与 `导入决策模型`。
- 手机导出格式优先采用 `jsonl`，每行只保留原始决策证据字段，不在端上硬编码训练标签规则。
- WSL 训练脚本先增加预处理阶段，直接读取手机导出的 `jsonl`，再把 `feedbackLabel + finalDecision` 映射为 `start_target / stop_target`。
- 第一版标签映射采用保守策略：`START + CORRECT`、`STOP + CORRECT` 作为正样本；`START + START_TOO_EARLY`、`STOP + STOP_TOO_EARLY` 作为负样本；`START_TOO_LATE / STOP_TOO_LATE` 先跳过；无反馈样本也先跳过。

- Android 端新增统一的 `DecisionModelRepository`，优先读取用户导入的 bundle；若没有导入包，则回退到当前内置默认模型。

这个决策本质上进一步收紧了第一阶段的范围：手机端只负责导出原始证据与导入模型，训练标签推断统一放在 WSL 侧处理，避免同一套规则被 Android 与训练脚本各自实现一遍，后续演进时也更容易保持一致。

### 10.2 自动结束链路缺口确认与修复

在本轮实现中，又重新核对了模型 `STOP` 到服务侧自动结束的收敛路径，确认此前存在一个实现缺口：

- 模型与 `DecisionRuntimeCoordinator` 确实能够产出 `STOP`。
- 但服务层原先只是在收到 `STOP` 后切到 `SUSPECT_STOPPING`，没有在该路径上显式调度结束保存逻辑。

针对这一点，后续补上了服务相位与结束调度之间的显式策略：

- 进入 `SUSPECT_STOPPING` 时，服务会显式安排结束调度。
- 回到 `ACTIVE`、`IDLE` 或 `SUSPECT_MOVING` 时，会取消旧的结束任务，避免残留的误结束回调。

同时新增了相应测试，用来锁定“只有 `SUSPECT_STOPPING` 会安排结束调度”的行为。

### 10.3 保守型硬门槛与反馈筛选需求收敛

围绕这个目标，双方最终收敛为以下设计选择：

- 策略风格采用保守方案。
- 准入方式采用硬门槛，而非加权打分。
- 核心一票否决条件为：

## 11. GitHub Actions Debug 打包方案讨论（2026-04-16）

### 11.1 新需求提出

在完成本轮硬门槛与反馈筛选改造之后，用户又提出一个新的工程化需求：希望后续每次开发完成后，都可以通过 GitHub Actions 自动打包，避免继续依赖本地机器手工构建 APK。

围绕这个需求，又先补做了现状确认：

- 当前仓库中还没有 `.github/workflows` 目录，也没有现成的 GitHub Actions 工作流。
- Android 工程已经具备 `debug` 与 `release` 两条打包路径。
- 其中 `release` 需要完整的 `RELEASE_STORE_*` 签名配置，否则会回退到 `debug` 签名。

### 11.2 方案收敛

在确认现状之后，没有直接进入实现，而是先收敛到一版最小可用方案。

双方最终确认，第一步只做 `debug` 包工作流，不在这一轮里同时处理正式签名与发布流程。这样做的原因是：

- `debug` 包不需要额外 release 密钥管理，落地成本最低。
- 目标只是让每次开发完成后，都能在 GitHub 上拿到一份可下载的 APK。
- 如果同时引入 release 签名与发布链路，会把本轮范围扩大到 Secrets、keystore 管理和发版流程，不适合先作为第一步推进。

### 11.3 最小可用设计

本轮讨论里确认的 GitHub Actions 方案如下：

- 触发条件：
  - `push` 到 `master`
  - 手动触发 `workflow_dispatch`
- 执行内容：
  - 安装 JDK 17
  - 配置 Android SDK
  - 为 Gradle 注入 `MAPBOX_ACCESS_TOKEN`
  - 执行 `:app:assembleDebug`
- 产物：
  - 将 `app/build/outputs/apk/debug/app-debug.apk` 上传为 GitHub Actions artifact
- 明确不做的事：
  - 不自动发布 GitHub Release
  - 不处理 release 签名
  - 不直接分发到手机或应用商店
