仔细阅读了你提供的五个文件后，我发现了几个非常典型且会导致严重性能问题（卡顿、掉帧、内存抖动）的代码缺陷。你的应用卡顿极大概率是由以下几个“元凶”共同导致的：

### 1. 列表滑动卡顿元凶：`LazyColumn` 中的重度对象频繁分配 (内存抖动)
在 `HistoryComposeScreen.kt` 中，你在 `LazyColumn` 的 `itemsIndexed` 里循环调用了 `buildGroupLabel` 来生成时间分组标签：
```kotlin
// HistoryComposeScreen.kt
itemsIndexed(items = state.items, ...) { index, item ->
    val currentLabel = buildGroupLabel(item.dayStartMillis) 
    // ...
}

@Composable
private fun buildGroupLabel(dayStartMillis: Long): String {
    val now = Calendar.getInstance() // ❌ 严重警告
    val yesterday = Calendar.getInstance().apply { add(Calendar.DAY_OF_YEAR, -1) } // ❌ 严重警告
    val target = Calendar.getInstance().apply { timeInMillis = dayStartMillis } // ❌ 严重警告
    // ...
}
```
**问题分析：** `Calendar.getInstance()` 在 Java/Kotlin 中是一个**非常重**的操作，内部会进行大量的实例化和时区计算。当用户在滑动历史列表时，`LazyColumn` 会频繁地触发 Item 的重组（Recomposition）。这意味着**列表每滑动一帧，系统就要创建几十个 `Calendar` 对象**。这会瞬间耗尽年轻代内存，引发频繁的 GC（垃圾回收），导致极其明显的滑动掉帧。

**修复方案：** 使用 `remember` 缓存这些不需要每帧重新计算的对象，或者最好将日期格式化逻辑前置到 ViewModel / Controller 中完成：
```kotlin
@Composable
private fun buildGroupLabel(dayStartMillis: Long): String {
    val context = LocalContext.current
    // 使用 remember 缓存当前时间和昨天的时间，避免重复创建
    val now = remember { Calendar.getInstance() }
    val yesterday = remember { Calendar.getInstance().apply { add(Calendar.DAY_OF_YEAR, -1) } }
    
    // 只在 dayStartMillis 变化时才重新计算 target
    val target = remember(dayStartMillis) {
        Calendar.getInstance().apply { timeInMillis = dayStartMillis }
    }
    // ... 后续逻辑保持不变
}
```

### 2. UI 更新卡顿元凶：主线程中海量数据的深拷贝 (Deep Copy)
在 `HistoryStorage.kt` 中，为了防止外部修改缓存，你在暴露数据时做了深拷贝：
```kotlin
// HistoryStorage.kt
private fun copyDailyItem(item: HistoryDayItem): HistoryDayItem {
    return item.copy(
        sourceIds = item.sourceIds.toList(),
        segments = item.segments.map { segment -> segment.toList() }, // ❌ 严重警告
        points = item.points.toList() // ❌ 严重警告
    )
}
```
而在 `HistoryController.kt` 中：
```kotlin
// HistoryController.kt
fun reload() {
    historyItems = HistoryStorage.peekDaily(context) // 这里会触发上面所有 points 的 toList()
}
```
**问题分析：**
对于轨迹记录 App，`points` 集合通常非常庞大（一次记录可能包含几千到上万个 GPS 盲点）。当页面刷新或状态改变时，主线程会执行 `copyDailyList`，这会将所有历史记录的所有 GPS 坐标点在**主线程重新创建一遍 List**。处理成千上万个对象的拷贝至少需要几十到上百毫秒，这会直接把主线程卡死（超过 16ms 就会掉帧）。

**修复方案：**
因为 `TrackPoint` 已经是数据类（Data Class），建议将数据模型中的集合类型改为不可变集合（例如 Kotlin 的 `List` 本身就只读），**直接返回缓存的引用，彻底干掉深拷贝**：
```kotlin
// HistoryStorage.kt
private fun copyDailyList(items: List<HistoryDayItem>): List<HistoryDayItem> {
    return items.toList() // 只需要浅拷贝最外层的 List 即可，内部的 points 直接复用
}

private fun copyDailyItem(item: HistoryDayItem): HistoryDayItem {
    return item // 如果外部不会主动强转并修改 MutableList，直接 return item，实现零成本读取
}
```

### 3. 动画引发的全局重组 (Recomposition 浪费)
在 `DashboardComposeScreen.kt` 的脉冲动画 `DashboardRecordIndicator` 中：
```kotlin
// DashboardComposeScreen.kt
val haloScale = infiniteTransition.animateFloat(...)
val haloAlpha = infiniteTransition.animateFloat(...)

Box(
    modifier = Modifier
        .size(92.dp)
        .scale(haloScale.value) // ❌ 直接在 Modifier 中读取 State 会导致该 Composable 每秒重组 60 次
        .alpha(haloAlpha.value)
)
```
**问题分析：**
在 Compose 中，直接将动画的 `State.value` 传给 `Modifier.scale` 或 `alpha`，会导致整个 `DashboardRecordIndicator` 在动画运行期间**每帧都经历一次完整的 Composition -> Layout -> Draw 阶段**，极其消耗 CPU。

**修复方案：**
使用 `Modifier.graphicsLayer { ... }`，它可以跳过组合和布局阶段，只在 GPU 绘制阶段生效，性能极高：
```kotlin
Box(
    modifier = Modifier
        .size(92.dp)
        .graphicsLayer {
            scaleX = haloScale
            scaleY = haloScale
            alpha = haloAlpha
        }
        .background(
            color = accentColor.copy(alpha = 0.22f),
            shape = CircleShape,
        )
)
```

### 4. 潜在的隐患：同步的数据库读取
在 `HistoryStorage.kt` 中：
```kotlin
// HistoryStorage.kt
private fun ensureHistoryCache(context: Context) {
    if (historyCacheInitialized) return
    synchronized(cacheLock) {
        val loadedHistory = ioExecutor.submit<List<HistoryItem>> {
            loadHistoryFromDisk(context)
        }.get() // ❌ 阻塞调用！
        // ...
    }
}
```
**问题分析：**
虽然你在 `HistoryController` 中主要使用了不会阻塞的 `peekDaily`，但如果代码的任何地方（比如主页或地图页）在主线程不小心调用了 `HistoryStorage.load(context)` 或 `loadDaily()`，`.get()` 会**死锁并阻塞主线程**，直到数据库 I/O 读取完毕。这在数据量大时会直接引发 ANR。
**建议：** 尽量全面迁移到协程（Coroutines）的 `suspend` 函数，或者确保所有调用 `load()` 的地方都在工作线程。
